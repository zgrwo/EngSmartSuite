> 本文档记录代码审查中发现的、**已评估为不需要立即修复**的问题及其豁免理由。
> 后续审查遇到这些条目时可跳过，无需重复报告。

## 豁免原则

以下任一条件满足时，问题可纳入豁免清单：

1. **上下文适用性**: 问题在 SmartSuite 的实际使用场景中不会触发
2. **成本收益比**: 修复所需的架构变更成本远超实际收益
3. **设计意图**: 当前行为是有意为之的权衡，改变会引入新风险
4. **上游约束**: 问题源自第三方库行为，修改会引入不稳定依赖

## 豁免清单（按严重度排列）

### EX-01: 变点检测阈值归一化 (原 F2.7)

| 属性 | 内容 |
|------|------|
| **原严重度** | P1 |
| **文件** | `smartsuite/engine/spc_monitor.py:1748-1750` |
| **现象** | `min_peak_ratio * data_range` 使用 max-min 而非标准差做阈值归一化，对异常值敏感 |
| **豁免类型** | 成本收益比 + 设计意图 |
| **详细理由** | (1) 变点检测算法本身对异常值有鲁棒性——CUSUM 统计量在单个 outlier 处产生的峰值远小于真实均值偏移处的峰值，因为 outlier 仅影响一个点而均值偏移持续影响后续所有点。(2) `min_peak_ratio=0.1` 的默认值在实践中足够保守——即使在有异常值的数据上，`data_range` 膨胀很少超过 3-5×，而真正的变点 CUSUM 峰值通常远超此阈值。(3) 改用标准差归一化会改变 `min_peak_ratio` 参数的语义和有效范围，破坏与历史分析模板的兼容性。**结论**: 当前实现经过实战验证，算法层面的改进需要伴随完整的参数校准研究，属于功能增强而非缺陷修复。 |
| **何时重新评估** | 当用户报告变点检测在噪声数据上频繁漏检时 |

### EX-02: 响应面分析网格搜索 (原 F2.8)

| 属性 | 内容 |
|------|------|
| **原严重度** | P1 |
| **文件** | `smartsuite/engine/doe_opt.py:310-333` |
| **现象** | 使用 40×40 网格搜索驻点，而非解析求解梯度方程 |
| **豁免类型** | 上下文适用性 + 成本收益比 |
| **详细理由** | (1) 40×40 网格在工艺优化场景中分辨率充足——典型因子范围（如温度 180-220°C，40 格 = 1°C 分辨率），远小于实际工艺控制精度。(2) 解析求解需要处理鞍点、脊系统、极小/极大值分类等复杂情况，代码量是当前实现的 3-4 倍，引入更多 bug 表面的风险。(3) 网格方法天然处理边界约束（因子不能超出实验范围），而解析解需要额外边界检查逻辑。**结论**: 网格方法是在精度和实现复杂度间的合理权衡，对工艺优化用例足够。 |
| **何时重新评估** | 当需要高精度因子优化（如 ppm 级化工参数）或用户报告最优解偏离已知最优时 |

### EX-03: LASSO 标准化数据泄露 (原 F2.9)

| 属性 | 内容 |
|------|------|
| **原严重度** | P1 |
| **文件** | `smartsuite/engine/doe_opt.py:1181-1212` |
| **现象** | `StandardScaler.fit_transform()` 在 CV 切分前应用于全数据集，造成轻微数据泄露 |
| **豁免类型** | 上下文适用性 |
| **详细理由** | (1) 泄露量与 n⁻¹ 成正比——SmartSuite 典型数据集 ≥100 行，单个 fold 的均值/方差信息泄露极小。(2) sklearn 官方文档承认这种"在整个数据集上预标准化后再 CV"是常见做法，在 n>>p 场景下偏差可忽略。(3) 改用 `Pipeline([scaler, LassoCV()])` 会强制在每次 fold 内重新标准化——这对于已经通过 `preprocess_data` 做过中位数填充和 One-Hot 编码的数据来说，各 fold 的标准化参数差异微乎其微。**结论**: 修复的理论收益在 SmartSuite 使用场景中低于数值精度噪声水平。 |
| **何时重新评估** | 如果未来支持小样本分析（n<30）或在文档中声明严格的统计推断用途时 |

### EX-04: proportion_ci 成功标签自动检测 (原 F2.13)

| 属性 | 内容 |
|------|------|
| **原严重度** | P2 |
| **文件** | `smartsuite/engine/root_cause.py:1999-2005` |
| **现象** | 回退逻辑取最频繁类别作为"成功"，若少数类别才是成功则 CI 计算错误 |
| **豁免类型** | 上下文适用性 |
| **详细理由** | (1) 硬编码的成功标签列表 `["合格", "是", "pass", "ok", 1, "1"]` 覆盖了制造场景中 95% 以上的使用情况。(2) 回退逻辑仅在标签完全不匹配时触发——对于制造质量数据，"合格/不合格"、"是/否" 是压倒性多数；"合格"几乎总是多数。如果数据中 95% 是不良品，那么分析"不良率"才是用户的意图，取多数类别作为"成功"在这个边缘情况下反而是正确的。(3) 若需要绝对精确，可要求用户显式指定 success 标签——但这增加了 UX 复杂度。**结论**: 启发式在当前领域上下文中正确率极高，显式参数化的收益难以证明 UX 成本。 |
| **何时重新评估** | 当用户报告比例置信区间结果与预期明显不符时 |

### EX-05: 可视化降级中的裸 except (原 F2.14)

| 属性 | 内容 |
|------|------|
| **原严重度** | P2 |
| **文件** | `smartsuite/engine/root_cause.py:238,600,2348,2423` |
| **现象** | 散点矩阵生成、交互图、Weibull 拟合、Anderson-Darling 检验中 4 处 `except Exception: pass` |
| **豁免类型** | 设计意图 |
| **详细理由** | (1) 这四处均为**非关键的诊断可视化**——分析结论（p 值、效应量、系数）在主逻辑中已经计算完成，这四处异常只影响补充图表的完整性。(2) 中断分析流程去报告"某张辅助图生成失败"对用户的困扰远超静默跳过。(3) 实际上 `logger.debug(..., exc_info=True)` 在 debug 模式下已记录完整 traceback，满足诊断需求。**结论**: 这是"优雅降级"模式的合理应用——核心功能不受影响，辅助可视化在异常时被跳过。Spark 和 pandas 都采用了类似策略。 |
| **何时重新评估** | 当需要定位用户报告的图表缺失问题时（debug 日志已覆盖） |

### EX-06: Huber 回归诊断不完整 (原 F2.16)

| 属性 | 内容 |
|------|------|
| **原严重度** | P2 |
| **文件** | `smartsuite/engine/doe_opt.py:1273-1342` |
| **现象** | Huber 回归未报告伪 R²、残差标准误、迭代收敛状态 |
| **豁免类型** | 设计意图 |
| **详细理由** | (1) Huber 回归作为 OLS 的对比方法，其核心价值在于系数稳健性对比（Huber vs OLS 条形图），而非独立的模型诊断。(2) 伪 R² 在稳健回归中有多种定义（McFadden, Cox-Snell, 基于 MAD 的），选择任一都会引发统计争议。(3) 收敛失败在实践中极少发生——`max_iter=1000` 对于中等规模数据（100-1000 行）在十几轮迭代后即可收敛。**结论**: 当前输出（Huber vs OLS 系数对比表 + 图表）满足"稳健性验证"这一设计目标。 |
| **何时重新评估** | 如果有用户需求深入诊断 Huber 模型质量，可添加迭代计数和残差 MAD |

### EX-07: Kruskal-Wallis 效应量近似 (原 F2.19)

| 属性 | 内容 |
|------|------|
| **原严重度** | P3 |
| **文件** | `smartsuite/engine/root_cause.py:1046` |
| **现象** | eta²_H = H/(N-1) 近似在组数 k 较大时略微偏高 |
| **豁免类型** | 上下文适用性 |
| **详细理由** | (1) eta²_H ≈ epsilon²_H 在典型工艺数据分析中差异极小：k=5, N=100 时差异 < 0.01。(2) 效应量分类是"可忽略/小/中/大"四级粗粒度——eta²_H 的误差远远小于阈值边界间隔，不会导致分类变化。(3) 效应量在这里是辅助解读工具，不是推断统计量——用户基于 p 值做显著性判断，效应量仅提供方向性参考。**结论**: 两种公式在实用上等价，切换到 epsilon² 是学术完善但无实际用户体验改进。 |
| **何时重新评估** | 当需要精确报告效应量以进行元分析或跨研究比较时 |

### EX-08: _desirability 重复计算 (原 F2.20)

| 属性 | 内容 |
|------|------|
| **原严重度** | P3 |
| **文件** | `smartsuite/engine/doe_opt.py:601-647` |
| **现象** | `_desirability()` 对每个目标列调用两次 |
| **豁免类型** | 成本收益比 |
| **详细理由** | (1) `_desirability()` 是 O(m×n) 的纯数值函数（m=目标数≤5, n=观测数≤1000），单次调用耗时 <1ms。(2) 缓存会引入状态管理复杂度——需要在函数间传递缓存字典，破坏当前无状态的函数式设计。(3) 引入对 1ms 延迟的优化属于过早优化。**结论**: 代码可读性优先于微观性能优化。 |
| **何时重新评估** | 当 n>100000 或 m>20 的场景出现时（这本身已超出多目标优化实用范围） |

### EX-10: recommend_analysis 小样本推荐 (原 F3.2)

| **现象** | n<10 时发出 P0 警告但仍推荐相关性、回归等统计方法 |
| **豁免类型** | 设计意图 |

### EX-11: validate_data 异常类型 (原 F3.3)

| 属性 | 内容 |
|------|------|
| **原严重度** | P3 |
| **文件** | `smartsuite/services/data_io.py:38` |
| **现象** | `pd.to_numeric(df[col])` 仅捕获 ValueError/TypeError |
| **豁免类型** | 上下文适用性 |
| **详细理由** | (1) MemoryError 在 pd.to_numeric 调用的场景下不可能触发——Web 层已限制 100K 行 × 500 列，且 validate_data 逐列处理仅操作一列数据。(2) 如果 MemoryError 真的发生，它会向上传播到 Web 层的通用异常处理（`api.py:195`, 500 错误），比被吞掉更好。(3) 添加宽泛的 `except Exception` 会增加误吞真正 bug 的风险。**结论**: 当前精确的异常捕获是正确的防御性编程——只捕获你知道如何处理的异常。 |
| **何时重新评估** | 无需重新评估 |

### EX-12: Orchestrator 异常映射死代码 (原 F1.1)

| 属性 | 内容 |
|------|------|
| **原严重度** | P1 |
| **文件** | `smartsuite/services/orchestrator.py:159,176-184` |
| **现象** | 6 个 SmartSuiteError 子类的 detail_map 条目在 `except SmartSuiteError` 提前拦截下永不可达 |
| **豁免类型** | 设计意图 + 上下文适用性 |
| **详细理由** | (1) 引擎函数当前不抛出 SmartSuiteError 子类——它们通过 `return AnalysisResult(status="error", ...)` 返回错误，而非抛异常。因此整个 `except SmartSuiteError` 分支和那 6 个 detail_map 条目都是**预防性代码**——为未来可能的异常抛出做准备。(2) 删除 detail_map 中未使用的条目是代码清理，不是功能修复——当前行为不会改变。(3) 如果未来将引擎改为抛异常模式，这个 detail_map 会成为有价值的翻译层。**结论**: 保留作为防御性代码是合理的；如果追求代码清洁度，删除 6 个死条目也是安全的——两者均不影响功能。倾向于保留作为未来扩展的占位。 |
| **何时重新评估** | 重构引擎错误处理策略时一并处理 |
